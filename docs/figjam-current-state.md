# FigJam 現況盤點

_盤點日期：2026-07-20_

目前的 [FigJam 使用者旅程](https://www.figma.com/board/OZG06ChjZGMaatLc2DDzQA)已畫出主要循環與權益生命週期，可供產品討論，但還不是能直接交給設計與工程的完整互動規格。

## 已有流程

### 發現與理解

封測完整權益 → 從短影音或公地進站 → 觀看 20 秒無登入序幕 → 接受紙上推演邊界。

接受者進入跟隨與造人；只想看市場方向的人返回盤勢研究頁。

### 跟隨與造人

先跟隨三名角色 → 選人生階段與背景 → 選研究傾向與誓言 → 選思考核心 → 確認人物底盤 → 五席觀測室。

模型 timeout 或格式錯誤時保存進度，內容不發布。

### 第一次觀測

進入五席觀測室 → 看同證異讀事件 → 展開「為什麼他這樣想」→ 查看象、證、界與來源 → 註冊並保存轉折 → 收盤鐘響。

資料不足時停止並說明。

### 每日收盤循環

收盤鐘響 → 角色事件卡 → 三層內心戲 → 選擇遞交反證、追問未知或安排角色對話 → 人物週卷。

真實交易日的紙上結果延後揭露。

### 每週、每季與公地

人物週卷 → 安排下週五席 → 八週一季 → 下一個交易週或公地。公地包含追蹤公開角色、回看信念與關係變化、匯出人物與觀察紀錄。

### 權益生命週期

封測提供完整權益、五席、模型與卷宗，不自動扣款。未來商業化才加入免費觀看、預設思考核心、盤勢 Pro、五席引導與一季循環。

未來權益到期後先完成當週，再把角色送入公地；私人筆記不公開。重新訂閱可召回最多兩名角色。

## 尚未畫進 FigJam 的層

目前缺口將由 pre-code review 排序。至少要補上：

- screen inventory 與每個 screen 的目的、入口、出口、資料需求和主要事件。
- 匿名、註冊、登入、跨裝置恢復、登出、刪除、匯出與同意版本。
- `truth_class`、來源、`available_at`、修訂、撤回與證據不足的可見狀態。
- perception packet 如何讓使用者看懂「角色當時看見什麼」。
- loading、empty、offline、stale、error、held-for-review 與 retry。
- 通知、時區、交易日與收盤前後的時間狀態。
- Web、SwiftUI、desktop、mobile、keyboard、screen reader 與 reduced motion 差異。
- 元件 state matrix、服務藍圖、資料責任與營運接手點。
- 從示範介面圖拆出的設計 tokens、元件、字型、角色資產與動效規格。

正式 Figma 介面稿開始前，FigJam 應先補足 P0 分支與錯誤狀態。畫面稿不能替代資料契約或法規邊界。
